万代CRM优惠券核销 — 中间系统方案

面向万代魂 Soul CRM 会员到店消费场景，以「收银员小程序 + 管理后台」构成的中间系统，打通优惠券核销与订单同步，且无需改造门店现有 POS。
版本 v2.1更新 2026-07-17状态 未评审产品经理 然谷

面临的问题

万代魂CRM向会员发放优惠券（兑换券、折扣券、满减券等），顾客持券到门店使用。门店POS系统无法直接对接万代CRM，需要在收银环节完成会员识别、验券和金额计算。

中间系统方案

自建中间系统，收银员通过手机小程序完成会员识别、验券、金额计算和条码生成，再通过POS扫码录入和收款。小程序负责对接万代CRM API，POS负责商品录入和支付结算。

方案优势

  • 无需改造POS：独立于POS系统运行，不影响现有收银流程
  • 系统校验：券的真伪、适用规则、金额计算由系统自动完成，降低人工出错率
  • 快速上线：中间系统可独立部署，先解决业务痛点
  • 可扩展：未来POS开放接口后，可平滑升级为直接对接

无系统支持时（现状）

  • 顾客出示万代优惠券，收银员无法验证真伪
  • 收银员手动计算优惠金额，容易出错
  • 券的使用规则复杂（等级限制、限购、有效期），收银员无法判断
  • 交易数据无法回传万代CRM，会员积分和等级无法更新
  • 退货时无法判断已核销券的处理规则

使用中间系统后

  • 扫码识别会员，小程序自动拉取会员信息和可用券
  • 系统自动计算优惠金额，生成POS端条码供扫描录入
  • 券适用规则、等级限购等由系统自动校验
  • 交易完成后异步同步至万代CRM，含重试机制
  • 退货规则内置系统，支持原路退回和券状态回滚

优惠券规则

  • 兑换券(EXCHANGE)：0元兑换指定商品，POS选"会员验券"收款
  • 折扣券(RATIO_OFF)：按比例减免，POS选"组合支付"
  • 满减券(AMOUNT_OFF)：达到门槛后定额减免
  • 多券可混合使用：系统自动计算每张券的优惠金额并汇总

券生命周期

每张券经历 4 个状态：available（可用）→ locked（锁定）→ consumed（已核销）/ unlocked（已解锁）。选券时锁定(lock)，交易完成时核销(consume)，交易取消（含退货）时解锁(unlock)。系统通过 lockTransactionId 全程追踪每张券的归属与状态。

组合支付分配

优惠合计 = "会员验券"金额（CRM承担），商品总价 - 优惠合计 = "现金/扫码"金额（顾客实付）。纯兑换场景无需组合支付，直接用"会员验券"收款即可。

异常与退货

  • 退货不返还券：已核销的券不予返还，只退顾客实付金额
  • 跨店退货：仅支持同一代理商下的快闪店之间
  • 同步失败重试：1min → 5min → 30min → 2h，最多4次
  • 数据回传：交易完成后异步同步订单，万代侧于 7 天后自动重算积分与等级

常见问题

收银员需要培训吗？
需要。收银员需同时操作手机小程序和POS两台设备，包括在小程序上扫码验券、查看操作指引、在POS上扫码收款等。初期建议安排专人带教，熟悉流程后再独立操作。
POS系统真的不需要改造吗？
不需要。本方案通过中间系统独立完成会员识别、验券与金额计算，不依赖 POS 侧的任何改造。代价是每笔交易需由收银员用扫码枪在小程序屏幕上多扫一次商品条码，再由 POS 完成录入与收款。
如果网络中断怎么办？
小程序会提示网络异常，交易暂停。已锁定的券不会丢失，网络恢复后可继续。同步失败的订单会自动重试最多4次（间隔1min→5min→30min→2h）。但中断期间收银台需暂停该笔交易，顾客需等待。
顾客退货时券怎么处理？
已核销的券不予返还，只退还顾客实际支付的金额。例如：原价¥580的商品用兑换券0元购买，退货时退款金额为¥0。跨店退货仅支持同一代理商下的快闪店之间。
新商品上架后多久能在小程序使用？
每日凌晨2:00自动同步万代商品数据。新商品还需要在后台手动完成"条码映射"（将万代商品编码与POS商品条码对应），映射完成后才能在小程序中生成条码供POS扫描。
数据安全如何保障？
所有API调用使用MD5签名鉴权，密钥仅存储在后台服务器。会员信息和券信息仅在交易期间有效，交易结束后自动清除，不在小程序本地留存。

目标、范围与非目标

明确本期要达成什么、覆盖哪些边界、以及本期明确不做的事项，避免范围蔓延。

本期目标（Goals）

  • 让门店在不改造 POS的前提下，完成万代会员识别、验券与优惠金额计算
  • 券核销与订单数据可异步回传万代 CRM，支撑积分与等级更新
  • 为收银员提供可照做的操作指引，降低人工计算与验券出错率
  • 支持首批快闪店试点快速部署上线

覆盖范围（In Scope）

  • 收银员小程序：扫码识别、会员/券展示、商品选择、POS 指引、交易完成
  • 管理后台：商品/条码映射、券核销记录、交易记录、同步监控、参数配置
  • 与万代 CRM 的会员、券锁定/核销/解锁、订单同步接口对接
  • 组合支付分配、多券混合计算、退货/取消的券状态回滚规则

非目标（Out of Scope）

  • 不改造门店现有 POS，不接管 POS 收款与开票
  • 本期不做 POS 直连（待 POS 开放接口后再规划）
  • 不做会员营销活动配置、券的发放与创建（归属万代 CRM）
  • 不做财务结算/分账系统，仅提供对账所需交易数据

非功能性需求（NFR）

功能之外的质量约束，是评审与验收的重要依据。

安全与合规

  • 传输：全链路 HTTPS(TLS1.2+)；签名由 MD5 升级为 HMAC-SHA256，MD5 仅在万代侧强制时兼容保留
  • 密钥：仅存后台服务端，支持定期轮换（建议 90 天），不下发小程序
  • 数据留存：会员/券明细交易结束即清除；交易与核销日志加密留存 180 天后归档
  • 个保：手机号等 PII 展示脱敏（138****8888），遵循《个人信息保护法》最小必要原则

性能

  • 扫码识别会员：P95 ≤ 1s，P99 ≤ 2s
  • 券核销/订单同步接口：P95 ≤ 800ms
  • 并发：支撑单店 ≥ 5 收银台，活动日峰值 ≥ 30 TPS
  • POS 条码扫描超时 5s，超时给出明确重试提示

可用性与容灾

  • 系统可用性目标 SLA ≥ 99.9%（月）
  • 万代 API 不可用时降级：允许"仅记录、稍后补同步"，券保持 locked 不误核销
  • 同步失败重试 1min→5min→30min→2h（最多 4 次），仍失败进入人工补偿队列
  • 关键故障 oncall 响应 ≤ 15 分钟；提供后台一键"手动重试同步"

审计与可观测

  • 全量操作日志/审计追踪：谁、何时、对哪张券做了锁定/核销/解锁
  • 核心指标监控：同步成功率、接口时延、核销量、失败告警
  • 调用链路 traceId 贯穿小程序→后台→万代，便于排障
  • 告警渠道：企业微信/短信，覆盖同步失败、时延超阈、异常核销

退货与取消交易闭环

补齐两条此前只在规则里提及、缺少流程的端到端场景。原型页可后续补充，本期先定义流程与接口时序。
退货退款流程
1
发起退货

后台/小程序按原订单号发起退货，选择退货商品

2
规则校验

已核销券不返还，仅计算可退顾客实付金额；跨店限同代理商

3
退款执行

门店按 POS 原路退回顾客实付部分

4
同步万代

调 trade/refund/sync 回传，积分等级由万代重算

中途取消 / 放弃交易（解锁券）
1
触发取消

顾客放弃或超时未完成，收银员点击"取消交易"

2
解锁券

调 coupon/unlock，locked→unlocked，券恢复可用

3
锁自动过期

未主动取消时，lockExpireAt 到期后系统自动解锁兜底

4
清理会话

清除本次会员/券缓存，回到首页

页面1：首页 / 扫码识别

交易入口，会员身份识别

页面职责

收银员打开小程序后的第一个页面。核心功能是扫描顾客的万代会员二维码，完成会员身份识别。同时提供手动输入手机号、退货退款等快捷入口。

交互规则

  • 扫描会员码：点击按钮调起摄像头扫描万代CRM会员二维码
  • 手动输入手机号：备用方案，二维码无法识别时使用
  • 今日交易：查看当班交易记录汇总

后端API调用

POST /fast-service/v1/0.0/api/member/get
参数类型必填说明
identify.userIdstring是扫码获取的会员标识
bizCodestring是固定值 bandai
requestChannelstring是POS
transactionIdstring是UUID幂等键

异常与特殊流程

扫码失败：非万代码/二维码损坏 → 弹窗提示"无法识别该会员码"
会员不存在：code=500000 → 提示"未找到该会员信息"
会员状态异常：status=INACTIVE/FROZEN → 拦截交易
退货退款：进入退货流程，需扫描原交易小票；券核销后不退回

设计原则

扫码是唯一的输入操作，零手动输入。会员信息在扫码后1秒内展示，不阻塞收银。敏感数据仅在当前交易中有效，交易结束后自动清除。

页面2：会员信息 + 可用券列表

展示会员身份、等级、可用权益

会员信息卡片

字段数据来源规则
姓名member/get → fullName脱敏显示：只显示姓，如"王**"
等级grade/get → gradeName7级体系：非会员/白/天/森/海/炎/极光
积分point/get → availablePoints当前可用积分
交易次数本地统计该会员在本店的历史交易次数

优惠券列表

POST /fast-service/v1/0.0/api/coupon/available/list

展示该会员所有状态为"可用"的优惠券：

券类型标签颜色展示信息
EXCHANGE(兑换券)绿色可兑换商品名称 + 有效期
RATIO_OFF(折扣券)黄色折扣比例 + 适用范围 + 有效期
AMOUNT_OFF(满减券)蓝色满减规则 + 适用范围 + 有效期
生日券红色优惠规则 + 生日月有效期

交互规则

  • 点击任意一张券 → 进入商品选择页面，自动筛选该券可兑换的商品
  • 点击底部按钮 → 展示所有可用券对应的全部商品
  • 返回按钮 → 回到首页，当前会员信息清除

业务规则

会员状态校验：只有 ACTIVE 状态的会员才能继续交易。
券有效期校验：已过期的券不会出现在可用列表中。
生日券特殊规则：仅当当前日期在会员生日月时，生日券才会出现。

页面3：商品选择 + 条码展示

小程序作为"商品目录+条码本"的核心页面

页面职责

根据会员持有的可用券，查询每张券关联的可兑换/可购买商品列表，校验规则后展示商品卡片。每件商品附带条码供POS扫码枪读取。

条码生成规则

核心要求：条码内容必须与POS商品库中的商品条码一致。
条码来源：①万代商品条码（如与POS一致）②本地维护的映射条码
展示要求：以可视化条形码图形展示，尺寸适配POS扫码枪读取距离（5-15cm）

规则校验（自动执行）

  • 折扣券：校验有效期 → 校验适用商品 → 校验最低消费 → 计算优惠金额
  • 满减券：校验订单金额是否达到门槛 → 计算减免金额
  • 抽赏券：校验购买时间段 → 校验预约门店 → 校验限购数量
  • 生日券：校验当前日期是否在生日月 → 校验适用商品

不可用商品处理

校验不通过的商品灰色展示，右上角标记原因（如"等级不满足"、"超出限购"）。不可用商品不生成条码，不可被选择。

商品类型与限制

类型限制条件
抽选品(1)需对应抽赏券，校验购买时段
限定品(2)校验会员等级 + 限购数量
TNS限定(3)TNS渠道限定
普通(4)无特殊限制
盲盒(5)随机款式，不支持选款

底部汇总栏

实时显示已选商品件数和优惠合计金额。优惠合计 = 所有券的抵扣金额之和。点击"确认"进入POS操作指引页面。

页面4：POS操作指引

引导收银员在POS上完成扫码录入和收款

金额计算逻辑

计算项公式示例
商品合计所有商品原价之和¥580+¥1,000+¥350=¥1,930
优惠合计所有券的抵扣金额之和¥580+¥100+¥30=¥710
顾客实付商品合计 - 优惠合计¥1,930-¥710=¥1,220
会员验券金额= 优惠合计¥710
现金/扫码金额= 顾客实付¥1,220

组合支付分配

会员验券金额代表CRM承担的优惠部分（¥710），现金/扫码金额代表顾客实际支付部分（¥1,220）。

纯兑换场景：当所有商品都是0元兑换时，无需组合支付。操作步骤简化为：①扫码 → ②选择"会员验券"收款方式 → ③确认完成。应收金额为¥0。

操作步骤（固定4步）

  1. 用扫码枪依次扫描商品条码（商品录入POS）
  2. 在POS上选择"组合支付"
  3. 输入会员验券金额和现金/扫码金额（按页面显示值）
  4. 完成收款后回到小程序点击"确认并同步"

后端API调用

点击"确认并同步"后，后台异步执行：

POST /v1/0.0/api/coupon/consume（券核销）POST /v1/0.1/api/trade/order/sync（订单同步）
核销时传入 lockTransactionId 完成券的最终核销。同步失败自动重试（1min、5min、30min、2h，最多4次）。

展示交易结果，支持开始新交易

展示信息

字段说明

交易时间POS完成收款的时间
商品数量本次交易的商品总件数
优惠金额所有券的优惠/抵扣金额合计
实付金额顾客实际支付的金额

数据同步状态

页面显示"数据已同步至万代CRM"是乐观展示——实际上同步是异步的。如果同步最终失败，后台会记录异常并在管理后台告警，不影响收银员继续下一笔交易。

后续操作

  • 开始新交易：清除当前交易所有数据，返回首页重新扫码
  • 万代后续处理：万代在订单同步7天后自动计算会员等级和积分变更

退货不返还券

重要规则：已核销的优惠券在退货时不予返还。退货退款只退还顾客实际支付的金额部分，券的核销状态不可逆。退货通过 /trade/refund/sync 接口同步。

仪表盘 - 功能说明

系统运营数据总览和待处理事项

页面职责

管理员打开后台看到的第一个页面。汇总展示当日核心运营指标，让管理员快速掌握门店运营状态。

统计卡片

指标数据来源刷新频率
今日交易笔数本地交易表 COUNT实时
今日交易金额本地交易表 SUM(实付)实时
今日核销券数本地券表 COUNT(consumed)实时
同步失败本地交易表 COUNT(FAILED)实时

数据看板

  • 最近交易：最近5笔交易及同步状态
  • 券核销分布：按券类型统计当日核销数量和占比
  • 系统状态：万代API连接状态、商品同步状态

待处理事项

  • 同步失败的订单（重试次数耗尽）→ 需人工点击"重试"
  • 未映射的新商品 → 引导跳转到"商品条码映射"页面
  • 锁定超时的券 → 引导手动解锁

页面职责

管理万代商品编码与门店POS条码的映射关系。这是整个系统正常运行的基础——只有映射正确，小程序生成的条码才能被POS扫码枪正确识别。

数据来源

POST /v1/0.0/api/mdm/product/query（待发布）POST /v1/0.0/api/mdm/productSaleRule/query（待发布）

万代商品主数据和销售规则通过每日凌晨2:00的自动同步任务拉取。

商品类型

编码名称说明
1抽选品需抽赏券兑换
2限定品有等级/限购要求
3TNS限定TNS渠道限定
4普通无特殊限制
5盲盒随机款式

注意事项

关键性：条码映射错误会导致POS无法识别商品，直接导致交易失败。新增映射后建议立即用小程序生成条码并用POS扫码枪验证。

页面职责

展示所有通过中间系统完成的交易记录。每笔交易包含完整的商品信息、优惠明细、券核销信息和万代同步状态。

同步状态说明

状态含义处理
已同步万代API返回success=true无需处理
同步中正在重试或等待重试自动处理
同步失败重试次数耗尽仍失败人工介入：点击"重试"
已退款该订单已执行退款通过refund/sync同步

重试机制

同步失败后自动重试策略：1分钟 → 5分钟 → 30分钟 → 2小时，最多4次。重试使用相同的 transactionId 保证幂等性。

页面职责

追踪每张优惠券的完整生命周期：available → locked → consumed/unlocked。用于排查券核销异常、处理锁定超时的券。

券状态机

状态触发时机对应API
available万代CRM发放给会员coupon/available/list
locked小程序选择该券并开始交易coupon/lock
consumed交易完成，确认核销coupon/consume
unlocked交易取消/中断，释放券coupon/unlock

lockTransactionId

锁券时由系统生成的唯一标识，贯穿券的锁定→核销/解锁全链路。核销时传入此ID，万代CRM据此确认是哪笔交易锁定的券。

异常处理

锁定超时：券被锁定后超过30分钟未核销，系统自动告警。管理员可手动点击"解锁"释放券。
退货不返还：退货退款后，已核销的券保持consumed状态，不返还为available。

页面职责

管理系统运行所需的所有配置参数，包括万代CRM接口凭证、同步策略、门店信息等。

签名机制

所有API请求需要MD5签名。签名算法：MD5(secret + key1 + value1 + key2 + value2 + ... + secret)，其中keys按字母排序。签名结果转大写。通过3个HTTP Header传递：X-Caller-Service、X-Caller-Timestamp、X-Caller-Sign。

同步策略

  • 商品同步：每日凌晨2:00全量拉取万代商品数据
  • 订单同步重试：1min → 5min → 30min → 2h，最多4次
  • 门店同步：管理员修改门店信息后手动触发

页面职责

集中展示系统运行中的异常和告警信息，帮助管理员及时发现和处理问题。

告警类型

级别类型触发条件
严重同步失败订单同步重试次数耗尽
严重券核销异常券锁定超时(>30min)
警告未映射商品商品同步发现新商品无法匹配
警告API连接异常万代API连续3次心跳失败
信息同步完成每日商品同步任务完成

处理操作

  • 同步失败：点击"重试"手动触发重新同步
  • 券锁定超时：点击"解锁"调用 coupon/unlock 释放券
  • 未映射商品：跳转到"商品条码映射"页面处理

告警通知

严重级别告警除在后台页面展示外，还可通过企业微信/钉钉/邮件推送给管理员。推送策略：立即推送（严重）、每30分钟汇总（警告）、每日汇总（信息）。
